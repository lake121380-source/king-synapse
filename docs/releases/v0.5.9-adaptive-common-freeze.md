# v0.5.9 — Adaptive Common Model Freeze

Tag: `v0.5.9-adaptive-common-freeze`

RFC: `docs/rfcs/RFC-011-adaptive-memory-common-model.md` — Status: **Implemented**

Prior milestones (all included in this freeze):

- `v0.5.1-memory-importance` — Importance skeleton
- `v0.5.2-memory-event-and-context` — Event kernel + AlgorithmContext closure
- `v0.5.3-benchmark-harness` — AlgorithmMetric + BenchmarkReport

## Architecture

**Adaptive Common Model is now frozen.**

The shared foundation every Phase 5 adaptive-memory algorithm consumes is now permanently stable:

- **Importance**: `MemoryImportance`, `ImportanceSignals`, `ImportanceSignal`, `ImportanceEstimator`, `NoOpImportanceEstimator`, `UniformImportanceEstimator`.
- **Event**: `MemoryEventId`, `MemoryEvent`, `MemoryEventKind`, `MemoryEventPayload`.
- **Event Stream**: `MemoryEventStream`, `NoOpMemoryEventStream`, `InMemoryMemoryEventStream`.
- **Context**: `AlgorithmContext<'a>` with fields `now`, `session_id`, `importance`, `events`. Trait-object surface closed at `v0.5.2`.
- **Metric**: `AlgorithmMetric` (11 IDs, `#[non_exhaustive]`).
- **Report**: `BenchmarkReport` (`#[non_exhaustive]`, `benchmark` + `metrics: BTreeMap<AlgorithmMetric, f64>`).

## Meaning

**Future work shifts from shared contracts to independent algorithms.**

What changes:

- Contract-first phase (RFC-001 through RFC-011) is complete.
- The next milestone is **RFC-012 Reflection Algorithm**, the first Phase 5 algorithm RFC.
- Development mode is now Algorithm-first. All algorithm work builds *on top of* RFC-011, never *modifies* it.

What does not change:

- No code changes in this milestone. `crates/core/src/adaptive/**` and `crates/eval/src/**` are byte-identical to `v0.5.3-benchmark-harness`.
- No test count changes: 112 tests continue to pass.
- Frozen benchmark baselines remain preserved:
  - `reference` dataset: `Recall@10 = 1.000`
  - `multihop` dataset: `Recall@10 = 0.600`

## Guarantee

**RFC-012–015 cannot introduce new shared models.**

Post-Freeze Rules (`docs/rfcs/RFC-011-adaptive-memory-common-model.md#post-freeze-rules`):

- **PF1** No new top-level shared types may be added under `crates/core/src/adaptive/` after v0.5.9. Additive `#[non_exhaustive]` variant extension of `ImportanceSignal`, `MemoryEventKind`, `MemoryEventPayload`, and `AlgorithmMetric` remains allowed.
- **PF2** Every Phase 5 algorithm's primary method MUST have the shape `fn method(&self, target: &T, ctx: &AlgorithmContext<'_>) -> Output`. Deviations require an ADR, not an RFC.
- **PF3** Downstream RFCs consume, never extend. RFC-012..015 MUST reference RFC-011 for `Importance`, `Event`, `Context`, `Metric`.
- **PF4** Algorithm RFCs MUST NOT depend on one another. All algorithms depend only on RFC-011. Cross-algorithm signaling flows through `MemoryEvent`, never through direct type dependencies.
- **PF5** `AlgorithmContext` never owns data. No new service handles (`&dyn Store`, `&dyn RecallEngine`, `&dyn PolicyEngine`, `&dyn Graph`, `&dyn LlmClient`) may be added.
- **PF6** Benchmarks use only the public API surface end users use.
- **PF7** Renaming any frozen type is breaking, regardless of behavioral equivalence.

## Freeze Boundary

Above `AlgorithmContext` → **frozen**. Importance, Event, Event Stream, Context itself, Metric, Report. Changes require an ADR and a `0.6.0` release.

Below `AlgorithmContext` → **open**. Reflection, Merge, Forget, Hebbian and their internal types under `adaptive/<algorithm>/`. This is where the next phase of engineering begins.
