# King Recall v0.4.39 - Store Integration Freeze

## Status

Store Integration is now considered **contract-frozen**.

This release closes P4.4. It defines the canonical persistence boundary for Phase 4 behavior modules and introduces the first Phase 4 milestone allowed to perform durable writes. It does not change the Recall contract, Memory Evolution contract, or any frozen behavior report.

## Execution Model

```text
ExecutionReport / ReflectionReport / HebbianExecutionReport
  -> StoreMutationDispatcher
  -> StoreMutationPlan
  -> StoreAdapter / PersistentStoreExecutor
  -> StoreExecutionReport
  -> StoreSink
```

Store Integration is the first Phase 4 area allowed to perform durable writes. All writes route through `StoreAdapter` or `PersistentStoreExecutor`; behavior modules must not call Store directly.

## Canonical Mutation Form

```text
StoreMutation
  InsertMemory
  UpdateMemory
  DeleteMemory
  ArchiveMemory
  UpdateEdge
```

Store does not know whether a mutation came from Consolidation, Reflection, or Hebbian. Persistent executors match on `StoreMutation` only.

## Freeze Rule

Future Store behavior must extend these interfaces rather than change them. Backend implementations must remain replaceable behind `StoreAdapter` / `PersistentStoreExecutor`. Breaking changes require ADR approval and a new architecture milestone.

Persistent executors must not introduce backend-specific report types and must not inspect behavior-module sources.

## Benchmarks

- `reference` remains `Recall@10 = 1.000`.
- `multihop` remains `Recall@10 = 0.600`.

## Next

**P4.5 - Adaptive Policies**

Policy layer (`ReflectionPolicy`, `HebbianPolicy`, `ForgetPolicy`, `MergePolicy`) sits on top of the frozen behavior chains and the frozen Store Integration boundary. Policies select or parameterize behavior; they do not redefine the execution model.
