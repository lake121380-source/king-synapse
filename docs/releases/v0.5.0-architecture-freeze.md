# King Recall v0.5.0 - Architecture Freeze

## Status

**Architecture: Stable.**

**Algorithm: In Progress.**

This is the first release where King Synapse commits to a stable public API and a compatibility policy. Phase 1 through Phase 4 are complete. All architectural contracts are frozen. From this release forward, the project transitions from **Contract-first** to **Algorithm-first** development.

## What v0.5.0 Freezes

Everything listed in `docs/API_SURFACE.md` as **Stable** is now covered by the compatibility policy defined in `docs/COMPATIBILITY.md`.

## Architecture Rules (Final)

Every capability follows the unified pattern:

```text
Trait
  -> NoOp
  -> Dispatcher
  -> Report
  -> Sink
```

Every layered call flows top-down:

```text
Policy
  -> Execution
  -> Storage
```

Every subsystem sits in the same stack:

```text
Recall Platform
  -> Working Memory
  -> Adaptive Memory
  -> Store
```

Every new capability introduced after v0.5.0 must follow these three shapes.

## Compatibility Policy

Starting with v0.5.0, King Synapse follows SemVer with pre-1.0 pinning:

- `0.5.x` — no breaking changes to stable APIs.
- `0.6.0` — first version allowed to introduce breaking changes; requires ADR approval.
- `1.0.0` — locks full SemVer promise.

Frozen benchmark baselines are covered by the policy:

- `reference` dataset: `Recall@10 = 1.000`
- `multihop` dataset: `Recall@10 = 0.600`

See `docs/COMPATIBILITY.md` for the full rules.
