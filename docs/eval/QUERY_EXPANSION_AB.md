# Query Expansion A/B: HyDE vs Category Expansion

Date: 2026-07-05
Dataset: 174 DMR no-rank samples (gold-answer chunk not in top-10 retrieval)
Embedding model: intfloat/multilingual-e5-base (same as system, via onnxruntime)
Expansion generator: deepseek-v4-flash, temperature=0, thinking disabled

## Purpose

Test whether query expansion (HyDE or category expansion) improves the
embedding similarity between the query and the relevant memory chunk, as a
proxy for whether it would improve retrieval recall.

This was motivated by the failure classification showing 73% semantic_gap
and 23% terminology_mismatch. The hypothesis was that expanding the query
with LLM-generated variants would bridge the semantic gap.

## Method

For each of the 174 no-rank samples, three query variants were generated:
1. **original**: the raw question
2. **hyde**: a hypothetical answer generated by DeepSeek ("answer this as
   if you are the person being asked, be specific and confident")
3. **category_expansion**: hypernym/hyponym terms generated by DeepSeek
   ("generate specific instances and related terms the answer might be about")

Each variant was embedded using the same multilingual-e5-base model the
system uses. Cosine similarity was computed between each variant and the
relevant chunk's embedding. A positive delta means the expansion brought
the query closer to the relevant chunk.

## Results

### Aggregate by failure type

| Failure type | n | Orig cosine | HyDE cosine | HyDE delta | HyDE better rate | Cat cosine | Cat delta | Cat better rate |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| semantic_gap | 127 | 0.800 | 0.785 | -0.015 | 15.8% | 0.787 | -0.013 | 24.4% |
| terminology_mismatch | 40 | 0.803 | 0.787 | -0.015 | 12.5% | 0.788 | -0.015 | 22.5% |
| chunk_boundary | 7 | 0.801 | 0.784 | -0.016 | 28.6% | 0.796 | -0.004 | 57.1% |
| **All** | **174** | **0.801** | **0.785** | **-0.015** | **16.1%** | **0.787** | **-0.013** | **24.7%** |

### Key observations

1. **Both expansion methods reduce cosine similarity on average.** The
   delta is negative across all failure types. This means expansion moves
   the query *away* from the relevant chunk in embedding space, not toward it.

2. **Category expansion is slightly less harmful than HyDE** (-0.013 vs
   -0.015 delta, 24.7% vs 16.1% better rate). This is consistent with the
   prediction that HyDE can hallucinate wrong answers (e.g., generating
   "dog" when the answer is "cow"), pulling the embedding in the wrong
   direction.

3. **The "better rate" is low for both methods.** Even category expansion
   only helps 24.7% of samples. For 75.3% of samples, expansion makes
   things worse.

4. **The original cosine similarity is already high (~0.80).** This is the
   most surprising finding. The relevant chunk already has high similarity
   to the original query, yet it does not appear in the top-10 results.
   This means the problem is not that the query-chunk similarity is too
   low, but that too many *irrelevant* chunks have even higher similarity.

## Analysis

### The hypothesis was wrong

The original hypothesis was: "the embedding model cannot connect the query
to the relevant chunk because of semantic gap, so expanding the query should
help bridge the gap."

The data shows this is not the case. The relevant chunk already has 0.80
cosine similarity with the original query. The problem is that ~10+ other
chunks have even higher similarity, pushing the relevant chunk out of top-10.

### The real bottleneck is ranking, not representation

If the relevant chunk has 0.80 similarity but doesn't make top-10, the issue
is that the embedding space is too crowded — too many chunks cluster around
0.80-0.85, and the relevant one gets outranked. This is a **precision**
problem (too many false positives with high similarity), not a **recall**
problem (the relevant chunk is reachable but not ranked high enough).

### Why expansion makes things worse

When you expand the query, you add more tokens. These tokens may match
irrelevant chunks *more* than the relevant chunk, because the expansion
introduces generic terms that appear in many chunks. So expansion increases
the similarity of irrelevant chunks more than the relevant one, worsening
the ranking.

### Implications for reviewer's index-time augmentation suggestion

The reviewer suggested augmenting chunks at index time (adding category
labels like "pet/animal companionship" to the "my cow keeps me company"
chunk). This A/B suggests this might also not help: the problem is not
that the chunk lacks the right keywords, but that too many other chunks
have high similarity. Adding labels to the relevant chunk would also
add similar labels to irrelevant chunks.

However, index-time augmentation differs from query expansion in one
important way: it only modifies the relevant chunk, not all chunks. So
it could still help if the labels are specific enough. This should be
tested separately.

## Conclusions

### Query expansion (HyDE and category expansion) does not help

Both methods reduce cosine similarity to the relevant chunk on average.
The hypothesis that "semantic gap → query expansion will help" is
**falsified** by this experiment.

### The real bottleneck is embedding precision, not recall

The relevant chunk already has 0.80 cosine similarity. The problem is
that too many irrelevant chunks have equal or higher similarity. The
fix should target **separating relevant from irrelevant chunks**, not
bringing the query closer to the relevant chunk.

### Recommended next steps (revised)

1. **Better embedding model**: A larger model (e.g., bge-large-en-v1.5)
   may produce sharper embeddings that separate relevant from irrelevant
   chunks more effectively. This is now the top recommendation.

2. **Reranker expansion**: The current reranker only processes pool=50
   candidates. If the relevant chunk is at rank 11-50, expanding the
   reranker pool could help. This should be tested.

3. **Index-time chunk augmentation**: Still worth testing, but with
   specific labels (not generic categories) to avoid the "all chunks
   get the same labels" problem.

4. **NOT recommended**: HyDE, query expansion, or category expansion.
   These are falsified by this experiment.

### What this means for the system

The system's retrieval bottleneck is not "can't find the answer" but
"can't distinguish the answer from noise." This is an embedding quality
problem that requires a better model, not a query transformation strategy.
